# Changelog

The v4rian git template's release history, in human readable form. Each section below corresponds to one `vYYYY.cycle.dev` tag landed on `dev` via the claim flow. Tags are the canonical seal of every release; this file is a companion that summarizes the substantive feature, fix, refactor, and docs commits behind each tag. Pure merge subjects and `chore(version)` plumbing commits are omitted; the full reachable history is always one `git log` away.

## Unreleased

Adds `required.commit-audit` as a sixth push-trigger required status check on PRs into dev so the `git-commit-audit` lens (atomicity, title format, body shape, bullet style, fabrication detection, file hygiene, branching, merge format) runs over every commit in the push range and blocks merge on any FAIL. The new wrapper at `.io.v4rian.how-git/ci/audits/push/required.commit-audit.sh` exec-delegates to the existing `audit-retry.sh` exponential-backoff loop (8 attempts, 5-to-300s backoff) with `--audit commit-audit`, and the worker at `.io.v4rian.how-git/scripts/audits/commit-audit.sh` resolves the push range from `GITHUB_EVENT_BEFORE`/`AFTER` or the origin/dev fallback, iterates commits, and applies the lens via `claude -p` per commit. Under `LIFECYCLE_TEST_MODE=1` the worker canned-passes without a network round-trip so the offline lifecycle suite exercises the wiring without API spend; in production it falls back to canned PASS when the `claude` CLI is unavailable so a missing client never blocks pushes. The `dev-required-checks.json` ruleset and `apply-repo-rules.sh` both bind the new context in addition to the auto-discovery in `bin/bootstrap-ci`, so all three binding sites stay in lockstep. A new lifecycle assertion pins the wrapper exec-delegation, the worker presence and executable bit, the `audit-retry` allowlist extension to include `commit-audit`, and the canned-mode functional path on both empty and non-empty ranges. The `RUNBOOK` is updated to list the new gate in the per-trigger inventory and to describe the `LIFECYCLE_TEST_MODE` plus missing-`claude`-CLI fallback semantics.

Extends the `PR #N` body-header prepend with the full GitHub PR URL after the number so the reference is click-to-open from any terminal that auto-links URLs. Both `claim-dev-merge.sh` and `claim-main-release.sh` now write the reference as `PR #N https://github.com/<owner>/<repo>/pull/N`; when `OWNER_REPO` is unresolvable (no remote, no `gh` host context) the reference falls back to the bare `PR #N` form so off-grid claims do not break. The lifecycle assertion that pins the body-header prepend is extended with two new greps asserting the URL append in both claim scripts so future drift breaks the suite rather than silently dropping the click-through.

Restores the merge commit body recap and prepends a `PR #N` reference right after the subject in every claim-flow amendment so dev and main merges stop landing as a spec subject plus blank body. Two coupled fixes: the `ci-dispatch.yml` matrix run step now binds `GH_TOKEN` and `GITHUB_TOKEN` to the workflow's `secrets.GITHUB_TOKEN` so `gh` calls inside every required script authenticate (the prior shape ran `gh pr view` unauthenticated, the call returned empty, and the script's `|| true` swallowed the failure); and `claim-dev-merge.sh` plus `claim-main-release.sh` now split the post-`gh pr view` check so an empty response logs loudly to stderr and falls back to the existing merge body, then unconditionally prepend a `PR #N` paragraph as the first line under the subject whenever the PR number resolved so the PR pointer is the first thing a reader sees before the recap section. Pool branches n58 through n64 all landed with the silent-empty-body shape before this fix; from n65 forward, the dev log carries either the PR's substantive recap or the GitHub-default merge body, with `PR #N` posted as the first body paragraph in either case. A new lifecycle assertion pins the ci-dispatch token bindings, the empty-body fallback in both claim scripts, and the `PR #N` prepend in both claim scripts so future drift breaks the suite rather than silently shipping the same regression.

Removes the force-push and close-and-supersede paths from the auto-rebase pass and codifies the how-git no-rewrite-history principle into the script's behavior. Previously, on a clean rebase against the new dev tip, auto-rebase force-pushed the rebased commits back onto the developer's pool branch (rewriting their history with new SHAs), and on a conflict it opened a separate `[RECONCILE]` PR and closed the original with a forwarding comment (closing the developer's PR on their behalf). Both paths violated the principle that the developer is sovereign in their `pool/*` sandbox and that the admin is the sole merge decider on dev. The rewrite makes the reconcile-branch path the only path: for every open pool→dev PR, the auto-rebase pass cuts a fresh `pool/nN.reconcile.<short>` candidate off the new dev tip, cherry-picks the pool's commits onto it (claude L1–L5 ladder on any conflict), pushes the reconcile branch as a CREATE (CI fires normally on the new ref without any PAT plumbing), and **comments on the existing PR** with the reconcile candidate and a recap. The original pool branch and the original PR are never touched. The admin reviews and decides: merge the reconcile candidate locally, tag the developer for changes, or close the PR to reject. The developer also owns the pool branch and can push their own resolution at any time before admin review; if they do, the reconcile candidate stays on origin as a receipt only. Receipts everywhere — original pool branch, reconcile branch(es), PR comment thread, vault on merge — all kept reachable for audit.

Splits the v4rian template into two explicit modes so a repo can choose whether the GitHub remote is the authoritative source of truth or the operator is. The new `connected` mode preserves the prior posture: `init-pool-merge` and `init-release` open PRs, the GitHub UI merges them, and the `claim-on-merge-dev` and `claim-on-merge-main` workflows drive the claim ceremony on the remote side. The new `off-grid` mode flips authority to the local admin: `git merge --no-ff` followed by `bin/finalize-pool-merge` closes pool to dev, and `bin/ship` closes dev to main, while the connected-mode entry points refuse to run and the always-present GitHub workflows no-op cleanly so a still-wired remote does not race local close-ups. Mode lives in `.io.v4rian.how-git/MODE`, defaults to `connected` on every fresh repo, and is flipped via the admin-gated `bin/set-mode` CLI. The claim ceremony itself now amends the merge subject to the canonical `[MERGE] pool/X into dev: vYYYY.X.Y` and `[MERGE] dev into main: vYYYY.X` spec forms directly from the source branch and version at claim time, replacing the prior validator-side title derivation that was prone to drift; the PR-open validator continues to handle body shielding, signature embedding, and the `opened-via-correct-flow` status check. The Repository Rules bypass list seeded by `apply-repo-rules` now includes both the Actions bot integration and the OrganizationAdmin role so org admins have a built-in emergency override without flipping enforcement off. The lifecycle test suite gains sixteen new tests covering MODE defaults, set-mode round-trip with admin-gate bypass under `LIFECYCLE_TEST_MODE=1`, doctor and status mode surfacing, wrong-mode refusal in either direction, claim-script no-op when off-grid plus CI both present, and a full connected to off-grid to connected flip cycle.

Promotes `bootstrap-ci` to the canonical one-shot repo setup by layering every repo-level setting that GitHub's `Use this template` button does not copy. The merge-commit defaults are PATCHed so the merge subject inherits the PR title (which carries the `vYYYY.X.Y` version stamp) and the merge body inherits the PR body (which IS the macro-tone TLDR of the change). The merge mode is locked to `--no-ff` merge commits only by disabling `allow_squash_merge` and `allow_rebase_merge`. The `delete_branch_on_merge` field is disabled so the claim flow can archive every merged pool branch to its paired `vault/` ref before GitHub would otherwise auto-delete it. Repo topics `v4rian` and `how-git` are added for discoverability. The required-status-check binding is extended from the lone `offline test suites` context to also include the three async audit contexts (`compile`, `coverage`, `runtime`) and the PR-open validator context (`opened-via-correct-flow`) on PRs into dev. Every new step probes current state via GET and only mutates on drift, so the whole run remains idempotent and re-invokable; `--check` reports drift without mutating, `--dry-run` prints every API call without executing.

Rebuilds the build-on-tag workflow around a canonical single-source `build.sh` and drops every distribution concern from it so the build half of the lifecycle is fully owned by this step and the distribution half can land cleanly in its own bundle. The workflow now triggers on both `push: tags: [v*]` (event-driven) and `workflow_dispatch` with a `tag` input (admin manual re-build of a historical tag), wraps the build in the same eight-attempt exponential-backoff retry shape the audit workflow uses, and asks `build.sh` with `BUILD_IDEMPOTENCY_CHECK=1` whether output for the target tag is already present at `build/artifacts/` before invoking the build body so re-runs against an already-built tag short-circuit cleanly. The `build.sh` script itself becomes a project-type dispatcher (Node via `npm install` + `npm run build`, Qt/CMake via a consumer-must-override banner, neither-marker via a template self-build text summary) with the output contract documented at the script header as `build/artifacts/` with `$TAG` embedded in the filename; consumer projects override the dispatcher body in their fork with their real recipe. Lifecycle suite gains five matching assertions (workflow shape with negative grep that no distribute script and no GitHub Release verbs sneak back in, build.sh template-self artifact, build.sh Qt-CMake override demand, idempotency probe round-trip) and drops the dead two-vs-three-segment shape heuristic and channel-detection tests that the new design replaces.

Adds the PR open stage of the v4rian lifecycle as a single source of truth shared by both pool to dev and dev to main flows. The new `.github/PULL_REQUEST_TEMPLATE.md` carries the macro body skeleton GitHub renders into the new-PR form, the new `validate-pr-open.yml` workflow fires on every pull_request opened, edited, or synchronize event into dev or main, and the new canonical `validate-pr-open.sh` script handles shielding (source branch shape plus actor admin check), title amendment via `bin/version`, body auto-fix via the shared library that the commit stage already uses, signature embedding, and posting the `opened-via-correct-flow` required status check. Two new local CLIs cover the developer entry points: `bin/init-pool-merge` for the pool to dev open, and `bin/init-release` for the dev to main release open with the PR body pre-filled from the CHANGELOG `## Unreleased` section. A small `[FIX]` in the shared auto-fix library corrects the title-body separator collapse so PR titles auto-derived from `git log --format=%s` render cleanly, and the lifecycle suite gains fifteen new tests covering the bundle (going from 81 to 96 total).

Moves the commit stage from a synchronous claude audit on the developer's laptop to a deterministic local auto-fix layer plus asynchronous CI checks. The new `pre-commit` hook scrubs em-dashes and en-dashes from staged files, the new `prepare-commit-msg` hook sources a shared library that rewrites messages in place (em-dash scrub, banned-word replacement, tag derivation from the staged diff shape, sentence cap on the description paragraph, third-person bullet rewrite), and `commit-msg` collapses to a cheap 8-100 char subject-length guardrail. The four claude audits that used to fire on `pre-push` (strategy, compile, coverage, runtime) move to a new `audit-pushed-branch.yml` workflow that runs on every pushed branch and surfaces `compile`, `coverage`, and `runtime` as required status checks on PRs to dev via a new `dev-required-checks.json` ruleset.
